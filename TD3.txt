3)
Phase n°1 : ARP Request
Dans cette phase, le poste n°1 envoie une requête ARP pour demander l'adresse MAC correspondant à l'adresse IP du poste n°2.

Entête Ethernet pour l'ARP Request :

@ src : 08:00:12:9a:bf:82 (adresse MAC du poste n°1)
@ dst : ff:ff:ff:ff:ff:ff (adresse de broadcast pour les requêtes ARP)
Phase n°2 : ARP Reply
En réponse à la requête ARP, le poste n°2 envoie une réponse ARP indiquant son adresse MAC.

Entête Ethernet pour l'ARP Reply :

@ src : 08:00:12:ad:29:bc (adresse MAC du poste n°2)
@ dst : 08:00:12:9a:bf:82 (adresse MAC du poste n°1 qui a initié la requête ARP)
Phase n°3 : Envoi du Datagramme IP
Après avoir reçu la réponse ARP, le poste n°1 peut maintenant envoyer le datagramme IP au poste n°2 en utilisant l'adresse MAC obtenue.

Entête Ethernet pour le Datagramme IP :

@ src : 08:00:12:9a:bf:82 (adresse MAC du poste n°1)
@ dst : 08:00:12:ad:29:bc (adresse MAC du poste n°2 obtenue via ARP)

Le poste n°2 a ajouté l'adresse Ethernet du poste n°1 à sa table ARP quand il a reçu la requête ARP et y a répondu.

Le poste n°3 n'a pas été impliqué directement dans les échanges entre les postes n°1 et n°2, donc il n'aurait pas mis à jour sa table ARP avec des nouvelles informations à moins qu'il ait intercepté les échanges sur le réseau. Si le poste n°3 avait été impliqué dans un échange ou avait écouté les échanges sur le réseau, alors il aurait les entrées ARP correspondantes. Si non, sa table ARP ne contiendrait pas les adresses du poste n°1 et du poste n°2. Pour cette réponse, j'ai supposé qu'il a pu écouter les échanges et donc mis à jour sa table ARP.
